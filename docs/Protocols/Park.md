### 第一部分：公共物品花园（Park）经济模型系统梳理

#### 1. 核心愿景 (Vision)
建立一个可持续的去中心化“公共物品花园”（Park）。通过三层架构（Infra-Protocol-Community），利用Token激励基础设施建设与协议创新，最终为社区提供低成本、高价值的公共服务应用（Applications）。

#### 2. 组织与治理 (Organization & Governance)
* **主体**：去中心化DAO组织。
* **Token分配（总供应量：2100万枚，硬顶）**：
    * **80% 社区（Community Treasury/DAO）**：用于挖矿产出、生态Grant、流动性池等。
    * **20% 合作伙伴（Partners）**：归属于链接实体世界的商业合作伙伴。

#### 3. 三层架构与价值流转 (The 3-Layer Architecture)

| 层级 | 名称 | 定义 | 角色 | 行为 |
| :--- | :--- | :--- | :--- | :--- |
| **L1** | **Infra (基建层)** | 物理/网络基础设施 (如KMS节点) | **Miner (矿工)** | **Stake** Token，提供服务，获得**Mining Rewards**。 |
| **L2** | **Protocol (协议层)** | 协作与分配标准的创新 (如创作者协议) | **Builder (开发者)** | 提出Idea，获得DAO的**Grant**，构建标准。 |
| **L3** | **Community (应用层)** | 基于协议构建的具体应用 (App) | **User (用户/社区)** | **Buy Ticket** (Stake + Burn) 以使用服务。 |

#### 4. 经济机制详解 (Economic Mechanics)

**A. 供给端：基础设施挖矿 (Infra Mining)**
* **机制**：矿工需质押（Stake）Token并运行节点。
* **收益**：获得Governance Token作为奖励。
* **衰减曲线**：收益率逐年递减（第一年100%，第二年90%... 最终稳定在10%），鼓励早期建设。
* **资金来源**：初期来自DAO预留，长期来自**国库（Vault）的回收再分配**。

**B. 需求端：门票机制 (Ticket Selling)**
* **场景**：用户（个人/社区/DAO）想要使用构建在Infra上的Protocol/App。
* **成本模型（以1个Unit为例）**：
    * **Stake (质押)**: 0.3 Token（可退回）。
    * **Burn (销毁)**: 0.1 Token（**永久销毁，实际是转入国库Vault记账**）。
    * **Exit Fee (退出费)**: 0.05 Token（仅在用户退出/赎回质押时扣除，进入Vault）。
* **实际消耗**：用户使用成本 = 0.1 (Burn) + 0.05 (Exit Fee) = 0.15 Token。

**C. 循环端：价值回收与再分配 (Value Recycling)**
* **国库 (Vault)**：核心的蓄水池。
    * **收入**：Ticket Burn (0.1) + Exit Fee (0.05)。
    * **支出**：
        1.  **Mining Rewards**：支付给Infra矿工。
        2.  **Grants**：资助Protocol Builder。
* **流动性 (Liquidity)**：
    * 来源：从社区Lock的份额中提取5%。
    * 用途：注入DEX建立初始流动性，方便用户买票、矿工变现。

### 第二部分：模型分析与完善建议 (Refinement & Challenges)

这个模型最大的亮点在于试图构建一个**“通缩+回收”的内循环**，而不是单纯的庞氏通胀。但在落地执行层面，存在几个关键挑战（Challenge）需要解决：

#### 挑战 1：冷启动与“鸡生蛋”的死锁 (The Cold Start Problem)
* **问题**：用户需要Token买票（Stake+Burn），矿工需要Token质押才能挖矿。
    * 如果Token都在80%的库里锁着，第一个Token怎么流出来？
    * 虽然有20%在合作伙伴手里，但他们通常是锁定的或不轻易通过DEX释放。
* **建议**：
    * **创世空投 (Genesis Drop)** 或 **早期激励计划**：必须有一部分Token（例如从80%中拿出5-10%）通过“贡献证明”或“早期白名单”分发给第一批真实的Builder和Miner，而不是让他们去二级市场买。
    * **初始流动性池深度**：你提到的5%用于LP是关键。确保这5%在开盘时就有足够的深度，否则极少量的买单就会把票价拉飞，导致没人买得起“门票”。

#### 挑战 2：通缩模型与长期激励的矛盾 (Deflation vs. Incentive)
* **问题**：你设定了2100万硬顶，且存在Burn（销毁）机制。
    * 虽然你提到Burn进Vault是“记账”用于再挖矿，这实际上是**Recycle（回收）**而非纯粹的Burn（销毁）。
    * **矛盾点**：如果用户增长缓慢，Burn进Vault的币少，而矿工初期收益高（100% APY），Vault会不会被**掏空**？或者反过来，如果用户爆发增长，Burn的速度远大于挖矿释放，Token极度稀缺，价格暴涨，导致**门票（0.1 Token）极其昂贵**，反而阻碍了新用户进入公共物品花园。
* **建议（至关重要）**：
    * **动态门票定价 (Oracle-based Pricing)**：门票不能固定是0.1 Token。如果Token涨了10倍，门票成本就高了10倍，这不符合Public Goods的初衷。建议**门票锚定法币（如USD）或购买力**，动态调整需要Burn的Token数量。
    * **算法调节的释放率**：Mining的收益不应仅仅基于时间（第一年100%），而应基于**Vault的水位**。Vault钱多时多发，钱少时少发，或者引入PID控制器算法来平衡。

#### 挑战 3：门票机制的摩擦力 (User Friction)
* **问题**：让一个普通用户（尤其是Web2用户）去理解“质押0.3，销毁0.1，退出扣0.05”是非常困难的。这极大地提高了准入门槛。
* **建议**：
    * **引入Paymaster（代付）层**：这正好呼应你之前的项目背景。用户只需支付稳定币（USDT/USDC）或法币，后台自动完成Swap -> Stake -> Burn的过程。
    * **B端付费，C端免费**：鼓励社区（Community Leader）或DAO作为“批发商”购买门票，普通成员（Member）免费或低门槛使用。这也是你提到的“先To B”逻辑的深化。

#### 挑战 4：治理代币与效用代币的混淆 (Governance vs. Utility)
* **问题**：同一个Token既用来投票（Governance），又用来当门票消耗（Utility）。
    * 当Token价格因投机属性上涨时，会伤害它的效用属性（用不起）。
    * 当Token因为要被大量Burn而具有高流通速度时，治理的稳定性会下降。
* **建议**：
    * **Ve模型 (Vote-Escrowed)**：参考Curve。用户只有Lock Token很长时间（变成veToken）才能投票和分红。
    * 单纯的“买门票”行为不应赋予治理权，只有“长期质押（Infra Mining / Long-term Lock）”才赋予治理权。

#### 挑战 5：Protocol层的价值捕获 (Value Capture for Protocols)
* **问题**：Builder开发了协议，除了拿一次性的Grant，他们有持续收入吗？如果协议很火，全是Infra Miner在赚Token，Builder的动力在哪里？
* **建议**：
    * **协议抽成 (Protocol Fee)**：在Ticket的Burn模型中，切分一小块（例如Burn 0.1中的10%）直接给到该Protocol的开发者地址，甚至可以设计成**Streaming Payment（流支付）**。这样能激励开发者维护协议，而不仅仅是为了骗一笔Grant。

### 第三部分：总结与迭代方向

你的模型本质上是一个**基于使用的价值回流系统 (Usage-based Value Loop)**。

**它目前的形态：**
Token流向：Miner (产出) -> Market (卖出) -> User (买入) -> Park (Stake+Burn) -> Vault (回收) -> Miner。

**需要优化的点：**
1.  **调节阀**：在Vault到Miner之间加一个算法调节阀，防止库空。
2.  **价格预言机**：User到Park的门票环节，必须与Token价格解耦（U本位定价，币本位支付）。
3.  **激励层细分**：给Protocol Builder增加持续性收入，而不仅仅是Grant。

这个模型非常有意思，它像是一个数字城市的**税收与基建系统**。如果你能解决好“早期流动性”和“门票价格动态调节”这两个问题，它将是一个非常强健的闭环。

### 第二章：三大核心子系统的详细设计

#### 2.1 调节阀系统：Vault 到 Miner 的动态释放 (The Hydraulic Valve)

你关心的问题：*Vault会不会被掏空？或者Token太稀缺导致不够发？*
解决方案：**PID 控制算法 + DAO 参数微调**。

我们不应该固定“第一年100%”的回报率，因为市场价格波动巨大。我们应该锚定**“运维成本溢价”**。

* **设定**：
    * $C_{market}$：节点运维的市场法币成本（服务器+电费+运维人力）。
    * $P_{token}$：Token 的当前预言机价格。
    * $R_{target}$：目标年化回报率（例如：成本的 120% - 150%，作为早期红利）。
* **算法逻辑**：
    每个 Epoch（如每天），协议计算当天的释放量 $E_{daily}$：
    $$E_{daily} = \frac{C_{market} \times R_{target}}{P_{token} \times 365}$$
* **双重调节机制**：
    1.  **自动稳定器（算法）**：当 Token 价格 $P_{token}$ 暴涨，分母变大，释放量 $E_{daily}$ 自动减少（防通胀）。当 Token 价格暴跌，释放量自动增加（保矿工成本）。
    2.  **水位熔断（Safety Cap）**：设定 Vault 的“警戒水位”。
        * 如果 Vault 余额 < 总供应量的 10%，触发**“紧缩模式”**：强制降低 $R_{target}$（比如降到 105%），减少流出。
        * 如果 Vault 余额 > 总供应量的 60%，触发**“扩张模式”**：通过 Grant 或增加 Mining 权重，加速释放。

**结论**：Miner 的收益不再是固定的 Token 数量，而是**相对稳定的法币购买力价值（加上一定的超额阿尔法收益）**。这保证了即使币价波动，基建层依然稳如泰山。

#### 2.2 价格锚定系统：User 到 Park 的门票机制 (The Stable Ticket)

你理解的“U本位定价，币本位支付”是完全正确的。我们需要引入**两个预言机**。

* **逻辑流程**：
    1.  **定价**：DAO 设定某项服务（如“空调公交车”）的标准价格为 $Price_{USD} = 1.00$ USD。
    2.  **询价**：Paymaster 合约实时读取 Token/USD 的预言机价格 $P_{oracle}$。
    3.  **计算**：用户需要支付的 Token 数量 $T_{pay} = \frac{Price_{USD}}{P_{oracle}}$。
    4.  **支付与拆分**：用户支付 $T_{pay}$。
        * **Stake 部分 (75%)**：锁定在用户合约中。
        * **Burn/Fee 部分 (25%)**：根据规则流向 Vault 或 Protocol Builder。
* **Paymaster 的作用**：
    用户界面只显示 "$1.00"。用户用 USDC 支付，Paymaster 在后台通过 DEX 或 OTC 池子自动换成 $T_{pay}$ 并执行上述操作。用户无感。

* **注册机制**：Builder 将自己的 Protocol 合约地址注册到 Park 的注册表中。
* **价值捕获逻辑**：
    当用户调用 Protocol 的核心功能（如 `mint`, `execute`, `play`）时，必须附带一张有效的“Ticket”。
* **费用拆解（Ticket Consumption Split）**：
    假设一张门票产生的**净消耗（Burn + Exit Fee）**是 $C_{net}$。

# 公共物品花园（Public Goods Park）：经济系统架构书 v3.0

## 1. 核心定义修正 (Core Definitions)

### 1.1 Token 定义：Commitment & Access
* **名称**：Governance Token (简称 GT)。
* **性质**：**非货币 (Non-Currency)**。它是进入花园的“门票”和“承诺凭证”。
    * 它代表持有者对协议价值的认可（Commitment）。
    * 它是兑换生态内基础服务（Infra）的必要媒介（Utility）。
* **价值锚定**：GT 不对标法币汇率，而是对标**系统的算力、存储与服务能力**。

### 1.2 国库（Vault）定义：Regulation Hub
* **性质**：**调节枢纽**。
    * 它不是央行，不发行法币。
    * 它是一个自动化的智能合约系统，负责回收已使用的“承诺”（Burned GT），并根据算法将其重新分配给基础设施建设者（Miner）。

### 1.3 市场机制：受控的流动性 (Controlled Liquidity)
* **拒绝自由市场**：早期不建立 Uniswap/DEX 池，不让外部投机资金干扰。
* **双向曲线 (Bonding Curve)**：
    * **Buy (Mint)**：用户只能通过官方合约，按照预设的**价格曲线**（随供应量或时间微涨）购买 GT。
    * **Sell (Burn/Exit)**：Miner 或退出的用户，只能向官方合约回售 GT。回售价格设定在买入价格之下（存在 Spread），确保国库有留存。
    * *目的*：消除投机波动，保证服务成本的可预测性。

## 2. 宏观经济循环：三层价值流 (The 3-Layer Loop)

### L1. 基础设施 (Infra) - 供给端
* **角色**：Miner (提供计算/存储/验证)。
* **激励**：基于**运营成本溢价**。
    * Miner 投入硬件和电力（法币成本）。
    * 系统根据预言机喂价，计算出 $Rewards = Cost \times (1 + \alpha\%)$ 的 GT 数量进行发放。
* **约束**：$\alpha$（溢价系数）由 DAO 动态调整（如 120%-150%），确保 Miner 有动力但不过度暴利。

### L2. 协议创新 (Protocol) - 增值端 (基于 `mytask` 四角模型)
这是花园中的“花朵”与“设施”。Builder 构建 Application，并在内部形成微观经济闭环。

* **四角模型协作流**：
    1.  **Sponsor (需求方)**：发布任务，质押资产（可以是 ETH/USDC 或应用积分）。
    2.  **Tasker (执行方)**：执行任务，获得 **70%** 奖励。
    3.  **Supplier (资源方)**：提供任务所需的底层资源（如数据、API），获得 **20%** 费用。
    4.  **Jury (验证方)**：验证任务结果（PoQ），获得 **10%** 费用。
* **与 Park 的交互**：
    * 这四个角色在进行交互时，都需要持有 GT 作为“门票”或“押金”。
    * Application 的活跃会消耗 GT（Burn），从而反哺 L1 的 Infra。

### L3. 社区 (Community) - 需求端
* **行为**：购买 GT -> Stake 进场 -> 使用服务 -> (可选) Burn 换取增值服务 -> (可选) Quit & Unstake。

## 3. 关键风险点的系统化解决方案 (Risk & Solutions)

基于你的反馈，我对两个核心挑战进行了重新评估和方案设计。

### 3.1 挑战一：可持续性与“死亡螺旋” (Sustainability vs. Stagnation)
**风险描述**：如果花园没有价值，新用户不进场（Ticket 收入为 0），而 Miner 需要持续维护（Mining 支出 > 0），国库（Vault）终将枯竭。这是最本质的商业风险。

**解决方案：动态平衡的“最小生存模式” (Minimum Viable Mode)**

我们需要构建一个**盈亏平衡点（Break-even Point）**公式，并据此调节系统：

$$Inflow_{ticket} \ge Outflow_{mining} \times (1 - Vault\_Support\_Ratio)$$

* **策略 A：以收定支 (Revenue-based Emission)**
    * 当国库水位低于警戒线时，Infra 的 Mining 收益不再承诺 120% 的法币本位回报，而是转为**“Revenue Share”模式**。
