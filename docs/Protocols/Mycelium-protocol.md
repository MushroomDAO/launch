# Mycelium 协议概述
赋能社区，自由个体，塑造数字未来。
通过持续迭代和维护**Infra**，到帮助社区构建可持续的**Protocol**服务，到参与具体落地场景的**Community DApp**，三个环节来帮助每个社区和个体，并提供一个松散的协作关系网络，类似于菌丝网络，自组织去中心，无许可。
Mycelium协议是一个协议，由Mycelium社区来通过DAO的方式维护和运转，任何社区和个体可以加入到上述三个环节，无论是参与公共物品建设，还是想借助协议，构建自己的社区可持续，或者共同借助Mycelium协议帮助下构建具体场景的闭环DApp。

## 0. Infra
我们把基础建设部分定义为Infra，所有infra自动加入Park协议，作为数字公共物品为所有社区和个体服务。
包括：
1. Layer1/2区块链：以太坊主网/Optimism主网/更多Layer2/部署关键合约，存储资产和关键proof hash on merkletree
2. IPFS/ARWeave：存储公开的图片和文件
3. Community Nodes：提供
- DVT：链下预言机+Guradian签名+Reputation计算器+遗嘱执行器+更多）
- BLS聚合+链上验证，底层BLS工具，未来替换为PostQuatumn签名和验证模块
- KMS，各社区为成员提供TEE级别的安全私钥存储和签名服务，支持社交恢复，支持上游合约账户
- AI-Doris，社区AI模块，支持隐私本地化个人部署和社区版本的高性能AI，对成员提供AI服务
4. 外部中心化依赖+多服务商冗余+未来迁移为去中心
- Alchemy RPC，未来替换为轻客户端以及社区联合运营的以太坊全节点
- Alchemy Bundler，未来替换为嵌入ETH客户端的底层API
- 其他冗余备份：QuickNode RPC，Candide Bundler等
- Alchemy 数据服务：索引链上数据，提供类数据库查询，未来替换为自索引的社区数据服务；考虑冗余其他服务商。
5. 社区节点两种运营模式
- Homemade，需要支持ARM TEE，需要可以连接CloudFlare反向Tunnel，基于Libp2p，节点通信，依赖原始节点提供同步服务从而启动
- 边缘计算的云服务，可选支持ARM TEE（国内外都有），正常部署和加入p2p网络即可，同样依赖原始节点同步和启动
6. DVT验证节点
- 默认协议启动是三个DVT（初始化三个社区提供），基础合约内默认最低三个联合签名才能通过BLS验证
- 合约可以多签配置门限数量，可以跟随节点的增加，提供一定冗余
- 所有DVT都是协议的核心计算模块，提供去中心化验证和计算，获得GTonen和未来规划的PNTs。
7. 个体客户端
- Chrome插件
- 本地PC客户端/私有本地服务端
- 手机客户端（iOS/Android)
- 无许可自部署的网页端
- AAStar部署的网页端

## Protcols
协议的简单定义：就是**一个公开透明的可迭代的协作约定**，无论是技术底层的通信协议，还是应用层Park协议，或者每个社区自己的协议。
[Park协议](Park.md)是核心协议，基于infra和协作机制构建。
