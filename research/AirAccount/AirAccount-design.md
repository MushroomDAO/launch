# AirAccount design

## 目标
完成论文，同时基于论文完成产品 prototype 的设计，进入开发并完成基础 demo。
### 完成论文
#### AirAccount Core 的论文
- 简洁的账户，扩展支持 7702
- 核心不可升级合约+Passkey+TEE 二次签名和验证
- 完整的验证流程：签名基于基于 Passkey+TEE，验证基于多节点 BLS 聚合多签 + 链上验证
- SDK 完整支持流程：签名，验证，恢复，遗嘱，Gas 支付，NFT 验证，社区积分打通支付 gas 等

#### 去中心化 Guardian+ 社交恢复的论文
- 核心账户职责：日限额，大额，遗嘱执行
- 去中心化 Guardian 职责：安全规则过滤，社交恢复验证
- 社交恢复：初始化分组多权重签名机制，恢复流程，验证流程
- 低成本，社区多层社会网络支持，多角色支持

#### SDSS 雨计算论文
- Libp2p 的基础通信机制+ARM 芯片硬件自发布
- 简单有效的共识机制和安全假设
- ENS API 和无需可注册 + 市场化竞争（Gas，验证，恢复，遗嘱等）
- 三个节点完成一个基础的雨计算网络，保持稳定基础上动态加入退出


### 完成 0.2 版产品设计和开发
一个开源无需可的增强安全的加密帐户解决方案，无缝衔接 Email 和 Web2 帐号，无感 Gas 支付，指纹 Passkey 负责意愿许可 + 可信社区节点 TEE 负责安全验证的双重签名 + 链上验证，随时随地搬家的社交恢复。
0.2 版控制范围：
- 支持 Email 和 Web2 帐号无缝衔接（Google，GitHub 和任意 Email）
- 支持指纹 Passkey 负责许可 + 可信社区节点 TEE 负责安全的双重签名和链上验证机制
- 支持无感 Gas 支付集成 SuperPaymaster
- 硬件钱包/社区节点运行在符合 ARM 芯片要求的硬件上，运行可信应用 TA

#### Interface 组件：Email 和 Web2 帐号无缝衔接
- 后台 postgresql 数据库
- 后端 Go，后面迭代为 Rust
- 完成绑定功能，添加数据库记录，使用节点签名加密关键数据，读取后需要节点解密
- AirAccount 机制：拥有多签和双签 + 日限额的合约账户机制

#### Security 组件：核心合约 + 指纹 Passkey 负责意愿许可 + 可信社区节点 TEE 负责安全验证的双重签名
- 核心合约支持多签替换 + 追加体外规则 + 分组多签/验证恢复机制
- 支持指纹 Passkey 负责许可 + 可信社区节点 TEE 负责安全的双重签名
- 支持随时随地搬家（支持一键搬家）
- 支持 EOA 的合约扩展，采用同一套服务机制：安全、恢复、Gas 等

#### SDSS 雨计算节点软硬件组件：TEE 硬件钱包+ARM 芯片的 TA
- 硬件钱包，跑在 TEE 上，基于 Rust
- 跑在和 Ubuntu 并行加载的安全芯片 OS 上的可信应用 TA
- 通过 专用 SDK 开发 TA，和 Ubuntu 内的 TEE 硬件钱包应用通信
- 主要是接受指定的 API 来签名和验证，其他的不做
- 性能要求不高，主要是安全，只做简单的几个动作，只对指定 ip 的指定 API 做响应
- 需要 Go 或 Rust 组装数据，例如获得链上的公钥等，然后通过专用 SDK 发送给 TEE 硬件钱包应用

#### SDSS 雨计算网络组件：安全共识 + 雨计算（非本项目范围）
- 相关资料：../SDSS/SDSS.md，以及目录下的其他文档
- 安全共识：Libp2p 的基础通信机制+ARM 芯片硬件自发布
- 雨计算：三个节点完成一个基础的雨计算网络，共识机制和安全假设，保持稳定基础上动态加入退出
- 经济模型：基于激励机制，节点可以获得 Gas 和挖矿 奖励

#### Gas 支付组件：无感 Gas 支付（非本项目范围）
- 相关资料：../SuperPaymaster/SuperPaymaster.md，以及目录下的其他文档
- 不在 AirAccount 系统内，依赖 SuperPaymaster
- 完成数据组装、两次签名后发送给 SuperPaymaster 的 SuperRelay，进行 gas 签名，然后提交上链
- 后续增加 NFT 验证
- 增加 ERC20 自动计算和代扣
- 增加社区积分打通支付 gas
