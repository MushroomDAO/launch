# Security Thinking

## 安全、合约和架构思考

D2FA的v0.1版实现（6月底基础完成，7月底完成demo测试）

本次思路是链下Validtor版本，链上合约等7212上线前再说。

组成部分：

1. Flutter版本的passkey功能demo
    1. 生成本地key pair后在validator进行注册public key，用来验证D2FA签名
    2. 收到Communit node发的交易信息后，能够对信息进行签名
    3. 然后发给validator，验证，发给Community node
    4. community node根据至少3个验证结果通过，则通过（我们设立5个测试节点）
2. Community node，去中心
    1. 调用AirAccount SDK，创建账户
    2. 私钥AES加密存储在TEE（本次做不到，加密用的随机数最好是硬件的TEE芯片来加密，目前先用固定的）
    3. 绑定hash和地址到node本地库
    4. 接受指令，验证，执行交易
    5. 独立运行validator服务，
        1. 接受注册
        2. 验签服务
3. AirAccount SDK
    1. 根据Email，创建账户，并发email（含快速计算出来的地址）给用户email
    2. binding hash（email的）和地址到node库
    3. 部署（这次分开？还是自动触发？）：部署模板到OP测试链
    4. 发行一个nft gas card 到该地址，储值100 U
    5. 发一个ENS（用户可以设置名字）到地址
    6. 默认空投1000PNTs到address
4. Racks
    1. 假想的购买场景
    2. 默认上架数字item 10个，例如咖啡token
    3. 默认空投1000PNTs到address
    4. 基于Flutter demo，扫描 Racks货架的item QR
    5. flutter demo 构造交易，提交给Community node 交易指令
    6. flutter demo 二次D2FA验证
    7. 交易通过
5. 如果私钥被偷或者被泄露，被破解
    1. 方案1: 合约模板要求必须有validator的验证后的community node签名

## 整体安全分析思考

1. 合约钱包，AA账户只有在合约内部内置了passkey的签名验证所需公钥，才有最大安全性
    1. 依赖：必须使用AirAccount提供的支持passkey内置的合约模板
    2. 依赖：7212或者D-validator，进行去中心化验证（链上或链下）
    3. 依赖：用户不会丢失passkey
        1. 目前我理解passkey的对应的keypair生成，是一次性的，如果本地丢失，又无备份
        2. 则永远丢失，而链上合约这种验证如果是内置，则丢失对于永远**锁死**
        3. 可以增加**D2FA(passkey）的更换**，涉及到用最高权限：Public Guardian
2. EIP7217在布拉格升级会内置EVM中，时间是2025年初
3. 我们先用D-Validator(也是社区节点）来做去中心化验证，超过5个社区，基本可以满足常规场景
    1. 因为常规黑客不会因为你个体一点点的资产，就有**动机出巨资**让超过一半的社区共谋来伪证你的签名从而获得你的资产
    2. 超过10个社区节点，基本社区节点本身为了**自身reputation**，也不会主动针对具体某人做伪证，声誉随时比某个个体损失还大
    3. 另外会有**slash机制**，以及**高阈值机制**（你的二次验证，必须是4个而不是3个验证），也增强了安全机制。
    4. 还有实时的**Alert机制**，和validator机制分开，当有任何资产异动时，都会通知你
        1. 如果通知收到而非你意图的操作，基本上只是死个明白
            1. 如果**事后通知**，说明你的2FA被盗
            2. 通知可在我们指定的bundler或者service合约进行**事前通知**，需要AirAccount机制
        2. 如果你是**SBT资产**, 则需要配合guraduan，来取回控制权（因为不可转移）
            1. 因为Sbt不会被转移，所以即便私钥丢失，依然可以通过social recovery恢复控制
    5. 恢复需要AAStar提供的**Public Guardian**服务，常规有5个匿名guardian安全性足以
        1. 复合验证方式，至少提供两种
            1. 自有社会关系做guardian
                1. 有限性不足
                2. 实效性不足
                3. 专业性不足
                4. 可以做1/5或者分组，5组中每组至少有一个，每个组社会关系隔绝
            2. 社区（真人）识别+事先约定的guardian来恢复
                1. 安全，因为是社区reputation和积分，不涉及大额资产，可以
                2. 社区是半个熟人组织，真人和共同经历的事情，基本可快速验证是否是本人
            3. 匿名guardian
                1. 社区内
                    1. 提供自己最初内容的一些见证记录的再次验证
                    2. 例如文档，视频，文本等等
                    3. 其他方式？
                2. 社区外
                    1. 提供预先记录的guardian地址（guardian必须匿名）和验证合约
                    2. 发起恢复申请
    6. 如果大额资产和完全匿名
        1. 组织资产，则需要多签（AAStar后续会提供服务）
        2. 个人资产
            1. 非匿名账户可以使用社区的guardian，验证你是你
                1. 论坛账户权限
                2. twitter账户权限
                3. Email权限
                4. 辅助账户（两个），例如游戏、EAS等等
                5. 持有其中部分，自己设置权重（被人不知道，社区可以投票后合约判定）
            2. 专业匿名guardian+每三个月一次演练恢复
            3. 紧急锁死签名（事先签名，一次性，时效性），提交后合约钱包锁死
            4. guardian联合解锁（等同于恢复）
            5. 使用隐蔽父子/双账户（两个账户父子或同等权限）
                1. 日常使用受限账户权限(例如日限额，总限额，时效性）
                2. 父账户受到严格保护（例如独立电脑/手机，日常断网）
                3. 高权限账户在收到Alert后点击紧急锁定24小时
                    1. 然后重置子账户
                4. guardian可以重置高级账户
