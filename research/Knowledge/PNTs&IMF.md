# PNTs协议设计 - 借鉴IMF机制的深度分析

## 原始设计概念

### Idea核心

建立一个协议，帮助成员达成目标（价值创造，价值传递，价值使用）从而Mint PNTs作为记录；PNTs的价值：来自于协议激励大家通过链上机制完成价值循环，通过公共池和动态调整。

PNTs类似于提款权：可以兑换各成员承诺的服务或商品，初始化分配来自于承诺和持续兑现的承诺，以及对应的动态修正信用。

### 协议核心机制

**Commitment Coin Offering (CCO)** - 承诺代币分发机制提供流动性

#### 初始化协议
a社区承诺提供稳定的A服务，发行aPNTs，例如发行1000 aPNTs对应了承诺的服务（三个月内）

#### 发行PNTs
PNTs是基于所有社区承诺的10%（锁定在合约），背书的协议流动性Token，对应了各成员的兑换承诺。基于交易额的行业平均利率上限来发行PNTs

#### 加入协议
b社区可以基于Pool（目前仅有a社区锚定的价值），来定价bPNTs对PNTs汇率

#### 协议收入
当用户使用aPNTs或者bPNTs进行消费且完成交易后，mint新PNTs，分配给各参与方（叠加于传统世界的交易），例如创建、传递和消费价值的各方

### 核心目标

保障PNTs的普及和价值保障（和IMF为美元服务，同时普惠各国类似）

## 具体建议

### 建议1: 分阶段实施(借鉴IMF历史)

```
Phase 1: 封闭测试(类似IMF 1945-1969)
  - 邀请3-5个头部社区
  - 标准化简单服务(如开发者时间、设计服务)
  - 手动治理,快速迭代
  - 目标: 验证核心机制

Phase 2: 限定扩展(类似IMF 1970-1990)
  - 开放至10-20个社区
  - 引入自动化定价(AMM)
  - 初步治理代币分配
  - 目标: 压力测试,发现边界

Phase 3: 全面开放(类似IMF 1990+)
  - 无许可加入(满足标准)
  - 完全链上治理
  - 跨链扩展
  - 目标: 规模化普及
```

### 建议2: 双代币模型

```
借鉴IMF"份额+SDR"双层结构:

Layer 1: 社区代币(aPNTs, bPNTs...)
  - 功能: 兑换本社区服务
  - 发行: 社区自主决定
  - 流通: 可在协议内外交易
  
Layer 2: PNTs(协议代币)
  - 功能1: 跨社区通用兑换权
  - 功能2: 治理权
  - 功能3: 协议收入分红
  
分离优势:
  ✓ 单一社区问题不直接影响PNTs
  ✓ 灵活定价(各社区代币独立浮动)
  ✓ 治理权与服务兑换权分离
```

### 建议3: 三层风险缓冲机制

```
Layer 1: 超额抵押
  社区锁定: 承诺价值的 150%
  (而非你提的10%,10%太低)
  
  理由:
    - MakerDAO: 150%抵押率
    - 应对价值波动
    - 违约时有缓冲

Layer 2: 保险基金
  协议收入的 30% → 保险池
  用途:
    - 社区违约时补偿用户
    - 极端市场波动时稳定PNTs
    - 类似IMF的黄金储备

Layer 3: 动态权重调整
  类似SDR篮子5年审查:
    - 每季度评估社区信用
    - 履约率低 → 降低权重
    - 自动触发,防止人为拖延
```

### 建议4: 清晰的价值锚定公式

```
PNTs价值计算(建议):

PNTs = Σ(社区i权重 × 服务i市场价)

社区权重 = f(
  锁定量: 40%,
  历史履约率: 30%,
  交易量: 20%,
  存续时间: 10%
)

服务市场价:
  - 优先使用链上价格(如Gitcoin赏金)
  - 辅助链下预言机(Chainlink)
  - 治理投票最终裁决(争议情况)

透明度:
  - 所有参数链上可查
  - 每日更新PNTs指数
  - 类似IMF每日公布SDR汇率
```

### 建议5: Mint机制的精细设计

```
问题: 持续mint → 通胀失控

解决: 借鉴央行的"泰勒规则"

Mint率 = 基准率 × (1 + 调整因子)

基准率: 
  行业平均利润率(如3%)

调整因子:
  + PNTs需求旺盛(价格>锚定值) → 多mint
  - PNTs需求疲软(价格<锚定值) → 少mint
  
通缩平衡:
  - 交易手续费的50% → 销毁
  - 违约惩罚 → 销毁
  - 社区退出时部分销毁
  
目标: 温和通胀(年化2-3%)
```

### 建议6: 渐进式去中心化

```
治理演进路径:

Year 1: 核心团队主导
  - 快速决策
  - 手动处理边界情况
  - 积累治理案例

Year 2: 多签+社区投票
  - 5/9多签执行
  - 社区投票建议性
  - 逐步放权

Year 3+: 完全链上治理
  - DAO合约自动执行
  - 社区投票有约束力
  - 核心团队退居顾问

原因:
  - 避免早期治理攻击
  - 给信任时间积累
  - 类似IMF前30年实际由美国主导
```

### 建议7: 明确的社区准入标准

```
最小可行标准(MVP社区):

1. 服务可量化:
   ✓ 开发工时: 可量化
   ✓ 设计交付: 可量化
   ✗ "咨询建议": 难量化(早期排除)

2. 链上可验证:

3. 最小规模:
   - 历史交易额 > 10万美元
   - 活跃用户 > 100人

4. 信用保证金:
   - 锁定USDC: 承诺价值的150%
   - 锁定期: 至少6个月
   - 违约扣除: 100%

类似IMF:
  - 早期仅44国(都是大国)

## 核心风险预警

### 风险1: 估值主观性

```
问题: "咨询1小时值多少PNTs?"
  - A社区: 1000 PNTs
  - B社区: 100 PNTs
  - 用户困惑,信任崩塌

解决:
  ✓ 初期限定标准化服务
  ✓ 参考市场价(Upwork/Fiverr)
  ✓ 用户评价系统
  ✗ 避免早期引入主观服务
```

### 风险2: 流动性陷阱

```
场景:
  用户持有1000 PNTs
  → 想兑换a社区服务
  → 但a社区服务已售罄(3个月排期)
  → PNTs变"废纸"
  → 恐慌抛售

解决:
  ✓ 强制社区保留20%产能给PNTs持有者
  ✓ 服务预约系统(链上排队)
  ✓ 跨社区替代服务推荐
  ✓ 二级市场(用户间转让PNTs)
```

### 风险3: 监管不确定性

```
潜在监管问题:
  - PNTs是证券?(Howey Test)
  - 跨境服务交易合规?
  - KYC/AML要求?

建议:
  ✓ 法律意见书(及早咨询)
  ✓ 限定司法辖区(如先美国/新加坡)
  ✓ 明确PNTs是"实用代币"而非证券
  ✓ 记录设计初衷(避免Ripple困境)
