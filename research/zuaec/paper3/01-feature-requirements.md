# ACN-Agent 功能需求清单（Paper 1/2/3 → 工程映射）

> **用途**：Paper 3 工程落地的功能需求总表。把 Paper 1（P/SC/N 框架）、Paper 2（ASM 协议）、Paper 3（五模块 + Tier 评测）的所有 feature 拆解为可开发的工程需求项，并标注优先级与所属里程碑。
> **底座**：PydanticAI（待研究确认后细化每项的实现方式 / gap）
> **最后更新**：2026-05-25

---

## 〇、设计基调（来自 talk-feedback.md，必须贯穿全部 feature）

1. **不是法外之地**：身份可溯（ERC-8004+KYC）+ 行为可审（ASM 签名+SC 链上）+ 违规可惩（SC 扣除+坏账）
2. **数据不出本地**：原始业务数据永不离开用户设备；只有向量化+DP 噪声的特征进入网络（LAAS）
3. **协议即固化的博弈论成果**：不追求全局最优，提供最优解方向引导；参与者遵循协议即受益
4. **公共模板 + 个性化权利**：协议有公共基础字段（MUST）+ 协作方自定义声明（可选）+ 双向确认
5. **relay 数据默认 3 个月自动清除**，加密存储，relay 可选长期保留

## 五、Tier A 最小可行 agent（MVP 边界）

**Tier A = 能参与 ACN 最基本协作的最小能力集**，对应 MAI 的 MUST 项：

```
Tier A MVP 必须具备：
  [M1.1] 文本输入解析
  [M2.1] 线性任务序列
  [M3.1] 工作记忆 + [M3.2] 基础情节记忆
  [M4.1] MCP 工具调用 + [M4.2] ASM 消息发送
  [M5.1] ASM 消息生成/解析 + [M5.2] 角色声明 + [M5.5] 双向确认
  [ID.1] ERC-8004 NFT + [ID.3] 账户 + [SEC.1] 私钥本地
  [SEC.2] ECDHE 加密 + [AUD.2] 消息签名溯源
  [NET.1] Nostr relay 连接
  [SCN.4] 至少一个演示场景协议
```

**这是 M1+M2 里程碑要交付的核心**——一个能注册身份、连接 relay、按一个场景协议完成首次协作的最小 agent。
