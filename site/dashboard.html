独创特性

Unique Features 21 items

我们不只是在构建产品——每一项都包含独创性的技术创新，解决数字公共物品领域真实存在的问题。

F1

Anti-Censorship Paymaster

论文已发布

问题：传统 Paymaster 依赖链下 API 签名服务器，存在审查风险——服务商可以拒绝为你签名。

方案：移除链下签名服务器，所有 Gas Sponsorship 完全在链上完成。链上不可停止，因此在签名环节实现了抗审查。

论文已发布于 arXiv（Journal 审核中）
F2

AirAccount — Email + 指纹注册

问题：MetaMask 安装、助记词备份、Gas 获取——Web3 入门存在无数障碍，大多数人习惯 Web2 的 Email 注册。

方案：基于 Email 注册 + 指纹认证，无需助记词、无需安装插件，即可生成 Web3 账户。与 ERC-4337 AA 深度整合。

F3

日限额硬顶 — Daily Limit Hard Cap

问题：传统钱包私钥被盗后，所有资产可被一次性转走。

方案：合约层面设置日限额硬顶。即使私钥被盗，每天最多损失限额内的金额，给你留出反应时间。

F4

意图与安全分离 — T1/T2/T3 分级签名

问题：所有交易使用同一签名权限，无法区分日常小额和大额转账的安全等级。

方案：日限额拆分为三层：T1（指纹即可）→ T2（需 Guardian 1 签名）→ T3（需 Guardian 1 + 2 签名）。额度由用户自定义，合约强制执行安全策略。

F5

TEE 硬件 KMS — 私钥永不泄露

问题：软件存储私钥容易被恶意软件提取。

方案：基于 TEE 安全芯片（ARM）实现在线 KMS。私钥只能签名、不能导出，即使物理获取设备也无法攻破硬件安全芯片的保护。

F6

社交恢复 + 社区多签 Safeguard

问题：私钥丢失意味着永久失去资产，传统助记词备份既不安全也不易用。

方案：创建账户时绑定两部手机（PassKey）作为 Guardian。私钥丢失时，两个 Guardian 可恢复账户（重新生成私钥，地址和资产不变）。万一 Guardian 也丢失，社区多签 Safeguard 作为最终备份——收取少量费用，通过人工审核协助恢复。Safeguard 无法主动发起恢复，只能响应你的请求。

F7

OAPD — One Address Per DApp

问题：所有 DApp 共用一个地址，链上行为可被归集追踪，隐私无保障。

方案：每次使用 DApp 时自动派生独立钱包地址。外部观察者只能看到一个新地址与应用交互，无法追溯到你的 ENS 或主钱包。基于加密派生，阻止数据归集和跟踪。

F8

Session Key — 限时限域授权

问题：每次链上操作都需签名，游戏、订阅等高频场景体验极差。

方案：签名一次，在特定合约 + 特定额度 + 特定时间范围内持续有效。适用于：链游（无需反复按指纹）、订阅自动付费、AI Agent 授权（经安全授权后，Agent 可代你采购、订阅、支付）。

F9

不可升级合约 — Non-Upgradeable

问题：可升级合约给了运营方后门——操纵价格、Rug Pull 都通过升级实现。

方案：账户合约部署后永久锁定，任何人（包括社区）都无法修改。封死攻击面，消除信任依赖。

F10

内置无感 Gasless

问题：传统支付 Gas 需要 Approve + Swap，手续费高、耗时长，且 Gas Token 不一定被供应商支持。

方案：跨社区无感支持——任何社区发行的 Gas Token 都可直接支付 Gas，无需 Approve、无需 Swap。用户完全无感。

F11

社区积分即 Gas Token

问题：社区积分缺乏实际用途，成员对积分无感知。

方案：社区贡献积分直接作为 Gas Token。社区通过 SuperPaymaster 一键部署自己的 Paymaster，积分立即拥有最基础的实用价值。

F12

四角色社区任务 — 先于 EIP-8131

独创

问题：去中心化任务协作缺乏标准化的角色分工和纠纷处理机制。

方案：四角色模型：任务发布者 · 执行者 · 资源提供者 · 仲裁者。EIP-8131 提出三角色，我们更早提出四角色（可查 GitHub 提交记录）。完成任务获得积分，积分可用于 Gas 支付。

F13

积分兑换柜台 — Redeem Counter

问题：社区积分需要流动性才能让成员有感知，光能当 Gas Token 还不够。

方案：Loyalty Points Redeem Counter——社区可上架礼品、服务、特权，成员用积分兑换。为积分提供流动性和价值感知，赋能社区自主运营。

F14

CityOS — 跨社区积分合规服务

规划中

问题：跨社区、跨行业使用积分需要合规合法，自行管理成本极高。

方案：社区的双生商业公司提供合规忠诚度积分服务：跨社区联合积分活动、创作有奖激励、社区开店。不仅提升老用户忠诚度，还能吸引新客户、获得社区志愿贡献。

F15

SuperPaymaster — AAStar 社区运维

问题：谁来运营公共的 Paymaster 服务？

方案：由 AAStar 社区运维的公共 SuperPaymaster 服务，去中心化、无链下签名服务器。作为整个生态的默认 Gas Sponsorship 基础设施。

F16

SuperPaymaster = Payment Master

问题：Web3 支付协议碎片化，MPP、MCP、ACP、x402、EIP-8004 各自为政。

方案：SuperPaymaster 不仅是 Gas Sponsorship，更是统一的 Payment Master——集成 MPP（Google）、MCP（Coinbase / Cloudflare）、ACP、x402、EIP-8004，几乎覆盖市面所有支付协议。

F17

Soulbound Token DID — 链上身份

问题：缺乏统一的链上身份和声誉聚合机制。

方案：注册协议成员时获得不可转让的 Soulbound Token，如同白板——记录你加入的社区、社区内 Reputation（由社区共识规则计算）和 General Reputation（由所有社区共同制定规则）。NFT 凭证加持，跨社区声誉可见。

F18

Agent 身份与支付 — AI Agent DID

问题：AI Agent 缺乏链上身份、支付能力和声誉积累机制。

方案：基于 EIP-8004 自动注册三合约 Agent，赋予独立身份。接通所有支付通道（x402、MPP、ACP、社区积分、Gasless），同时积累 Reputation。

F19

社区 NFT — 一键发放凭证

问题：发行 NFT 门槛高——合约部署、Metadata 托管、跨平台兼容等障碍重重。

方案：兼容 POAP、OpenSea 等协议，支持多种用途：见证、凭证、荣誉、参与证明、服务、礼物、商品 Item。更简单的发放与查看方式。

F20

自托管 Paymaster — Clone & 自运营

问题：不想依赖 AAStar 社区运营的 SuperPaymaster，希望完全自主掌控。

方案：完全开源，Clone 代码后自行部署运营。同样去中心化、无链下签名服务器，与 AAStar 零关联。真正的无许可公共物品。

F21

跨社区开放商店 — Open Shop

规划中

问题：社区内的兑换局限于社区内部，缺乏跨社区的开放市场。

方案：面向全网开放的跨社区商店，打通社区间的价值流通。（设计规划中）
